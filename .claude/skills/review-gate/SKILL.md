# 검토 게이트 (완료 승인 절차)

## 절차
1. **수집**: JQL `project = KB AND status = "검토 중"` 조회 (jira 스킬 규격, 키+작업명 병기).
2. **이슈별 대조**:
   - 본문의 **DoD 항목**을 기준으로 삼는다 (DoD 없으면 반려 사유).
   - 관련 코드 확인: FE = `/Users/yejinkim/dev/kfood/kbap-fe`, BE = `/Users/yejinkim/dev/kfood/kbap-server`.
     최근 커밋(`git log --oneline -20`)과 해당 파일을 읽어 DoD의 **관측 가능한 결과**가
     실제로 충족되는지 판정. "구현했다" 주장이 아니라 코드/실행 근거로.
   - 진행로그 댓글의 블락 항목 확인(미해소 블락 = 완료 불가).
3. **안전 크리티컬 추가 관문**: 위험도·스캔·게이팅 관련 이슈는 DoD 외에 **false-safe 관점**을
   한 번 더 본다(불확실→unable/caution 강등 경로, 빈 프로필 safe 금지 — 헌법 III).
4. **판정 및 전환** (쓰기는 항상 사용자 승인 후 — jira 스킬 원칙):
   - **통과** → `완료`(전환 41) + 필요 시 BE-논의 플래그(jira §7)
   - **미달** → `진행 중`(전환 21)으로 반려 + 댓글로 미달 DoD 항목만 1~3줄
5. **보고**: 표로 — `키+작업명 | 판정 | 근거 1줄`. 반려 건은 후속 프롬프트(`fe-prompt`) 제안.

## 원칙
- 리뷰어는 구현 세션과 **다른 컨텍스트**여야 한다(이 레포 세션이 그 역할). 구현 세션의
  자기 완료 선언을 그대로 믿지 않는다.
- 실기기 확인이 필요한 DoD(예: 실물폰 OCR)는 코드 리뷰로 대체하지 않고 사용자 확인을 요청.

## 브릿지 보고 검토 후속 (2026-07-21 추가)
브릿지(REPORTS.md) 검토 = FE가 뭔가 끝냈다는 뜻이므로, 검토 통과 시 **반드시 QA.md를 동기화**한다:
- 해당 수정의 실기 확인 항목을 QA.md(해당 Q 또는 상단 배너)에 추가/갱신 — 환경(OTA 채널·강제종료
  2회·플랫폼)까지 명시. 새 기능이면 Q-항목 신설.
- 이미 있던 Q 항목의 상태·환경 줄이 낡았으면 현행화 (예: "OTA 후" → "OTA 발행됨, 지금 가능").
- 마지막 보고에서 **QA.md에 뭐가 바뀌었고/추가됐는지 예진에게 명시적으로 알린다**
  (예진의 다음 행동 = 폰 QA이므로, 문서만 고치고 침묵 금지).
