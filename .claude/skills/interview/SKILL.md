# Interview — 착수 전 요구사항 잠금

(원전: VelvetShark /interview 커맨드 — 우리 워크플로우에 맞게 조정)

## 절차

1. **읽기 먼저**: 인자로 받은 스펙 파일(있으면)과 관련 정본(spec.md·스웨거·기존 기획안·
   해당 코드)을 먼저 읽는다. **코드/문서로 답할 수 있는 건 절대 묻지 않는다** — 직접
   확인하고 넘어간다.
2. **인터뷰**: AskUserQuestion으로 비자명한 것만 묻는다 — 기술 구현, UI/UX, 엣지 케이스,
   트레이드오프, 완료 기준(acceptance criteria). 설계 트리의 가지를 하나씩 내려가며
   **결정 간 의존성 순서대로** 해소한다. **모든 질문에 내 추천안을 첫 옵션으로** 제시한다.
3. **끝까지**: 스펙이 구현 가능해질 때까지 계속한다(한 번에 몰아서 4개 이하씩,
   답이 다음 질문을 갈라치는 구조로). 사용자가 "이제 됐다" 하면 중단.
4. **확정 전 요약**: 최종 스펙 아웃라인을 요약해 확인받는다.
5. **산출**: 파일 인자가 있으면 그 파일을 완성 스펙으로 갱신, 없으면
   `specs/001-personalized-menu-mvp/` 아래 새 기획안 파일로 저장. **코드는 쓰지 않는다** —
   구현은 확정 스펙을 fe-prompt로 브릿지에 발주하는 별도 단계.

## 우리 레포 결합 규칙

- 데이터 모델·API 전제는 반드시 정본(스웨거·spec.md) 실확인 후 질문 — 기억으로 전제 금지
  (선례: 종교·식이 코드 드리프트).
- 헌법 제약(이모지 금지·i18n·위험도 4상태·reader/place 축)에 걸리는 답이 나오면 그 자리에서 지적.
- 확정 스펙에는 "결정·근거·제외한 대안"을 남긴다 (주문카드 브리프 형식).
