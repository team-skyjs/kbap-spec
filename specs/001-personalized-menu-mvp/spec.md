# Feature Specification: 개인화 한식 메뉴 안전 안내 (Personalized Korean Menu Safety Guide) — MVP

**Feature Branch**: `001-personalized-menu-mvp`

**Created**: 2026-06-15

**Status**: Draft

**Input**: docs/최초-기획서.md 기반 1차 MVP 범위. 헌법 v1.0.0 준수.

---

## 개요 (요약)

한국을 여행/거주하는 외국인이, 자신의 알러지·종교·식이 제약을 등록해 둔 상태에서 **한식
메뉴판을 촬영**하면, 단순 번역을 넘어 **"내가 먹어도 되는가"를 안전/주의/위험으로 안내**받고,
음식별 정보와 같은 국적 사용자들의 리뷰를 확인하며, '주의' 성분은 **사장님께 한국어로 직접
확인**할 수 있게 돕는 모바일 앱. 정보는 '식당'이 아니라 **'음식' 단위**로 모인다.

이 명세는 1차 MVP 범위만 다룬다. (제외: 가공식품/음식 직접 인식, 커뮤니티 게시판, 푸시
리마인드, 지도/식당 엔티티)

## Clarifications

### Session 2026-06-15

- Q: 스캔된 메뉴 항목을 어떤 기준으로 동일 음식으로 묶어 집계하는가? → A: 큐레이션 DB의
  표준 음식 카탈로그(canonical catalog)를 음식 ID 단일 기준으로 사용. 스캔 항목을 표준 음식으로
  정규화해 집계하며, 카탈로그에 없으면 '미등록 음식'으로 처리.
- Q: 리뷰 평점 척도는? → A: 5점 별점(1~5, 정수). 전체/같은 국적 평점은 이 척도의 평균으로 집계.
- Q: 랭킹 등급 산정 기준은? → A: 복합 점수(리뷰 수 + 리뷰한 음식 다양성 + 스캔 활동, 가중치
  적용). MVP는 이 세 요소만 사용하며, 2차에서 커뮤니티 활성도를 가중 요소로 추가 예정.
- Q: 계정 삭제 시 민감정보(알러지·종교) 처리는? → A: 개인·민감정보는 즉시 완전 삭제하고,
  작성한 리뷰는 익명화하여 보존(음식 평점 집계 유지).
- Q: 큐레이션 DB에 없는 '미등록 음식' 처리는? → A: 번역은 표시하되 위험도는 '판정불가'
  상태로 두고 "안전 판정 불가, 사장님께 확인" 경고를 노출. 미등록 음식은 백엔드에 로그로
  기록(추후 카탈로그 보강에 활용).
- Q: '홈' 탭의 내용은? → A: 개인화 허브 — (1) 최근 스캔/조회한 음식 + 리뷰 유도, (2) 프로필
  기준 안전도가 표시된 추천 음식, (3) '메뉴판 스캔' CTA. 새 엔티티 없이 기존 스캔/음식/평점
  재사용. (기획서의 '30분 후 리뷰 리마인드'(2차 푸시)의 인앱 자리도 여기에 마련)

### Session 2026-06-22

- Q: 카메라 메뉴판 번역 결과 표시 방식은? → A: 촬영 이미지 위 **오버레이형**(파파고/구글렌즈식)
  + 각 항목 우측 위험도. 추가로 **원본↔번역 오버레이 토글** 버튼 제공(FR-008, FR-037).

### Session 2026-06-29

- Q: MVP의 reader 언어(지원 언어) 범위는? → A: **MVP가 9개 언어를 처음부터 모두 지원**한다(출시형).
  지원 목록(BCP-47): **`en`(영어), `zh-Hans`(중국어 간체), `zh-Hant`(중국어 번체), `ja`(일본어),
  `vi`(베트남어), `id`(인도네시아어), `th`(태국어), `ru`(러시아어), `es`(스페인어)** — 한국 인바운드
  관광객 다수 언어 세트. 정적 UI 문자열은 9개 i18n 리소스로 제공하고, 동적 콘텐츠(메뉴명·리뷰·
  사장님 문구 등)는 BE 서버측 번역으로 readerLanguage에 맞춰 제공한다. 폰트는 라틴+CJK+태국+
  키릴 스크립트를 모두 커버해야 한다(글자 깨짐 0). place 언어는 MVP=ko 유지.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 식이 프로필 온보딩 (Priority: P1)

처음 앱을 설치한 외국인 사용자가 회원가입을 하고, 자신이 못 먹는 음식(알러지·종교·비건/채식),
국적, 사용 언어, 매운맛 허용도, 관심 한식을 등록한다. 이 프로필이 이후 모든 개인화의 기반이
된다.

**Why this priority**: 개인화(헌법 IV)가 제품의 본질이며, 위험도 안내(US2)는 이 프로필이
없으면 동작할 수 없다. 가장 먼저 존재해야 하는 토대다.

**Independent Test**: 신규 사용자가 가입 → 알러지/종교/식이 정보 입력 → 프로필이 저장되고
다시 조회되는 것으로 단독 검증 가능. (이 단계만으로도 "내 식이 정보 관리" 가치를 제공)

**Acceptance Scenarios**:

1. **Given** 앱을 처음 실행한 사용자, **When** 이메일 인증 또는 애플/구글 계정으로 가입하고
   닉네임·국적·언어를 입력하면, **Then** 계정이 생성되고 온보딩 다음 단계로 진행된다.
2. **Given** 가입 직후 사용자, **When** 알러지·비건/채식·종교 기반 못 먹는 음식을 구체적으로
   선택/입력하면, **Then** 해당 제약이 프로필에 저장된다. 이 단계는 **건너뛸 수 있으며**, 건너뛴
   사용자에게는 개인화 위험도 대신 '주의/판정불가'만 표시되고 추후 입력을 유도한다(false-safe 금지).
3. **Given** 필수 입력을 마친 사용자, **When** 매운맛 허용도(0~10)와 관심 한식을 선택하면(선택),
   **Then** 저장되고, 건너뛰어도 온보딩이 완료된다.
4. **Given** 매운맛 단계 선택 화면, **When** 사용자가 각 단계를 보면, **Then** 외국인이 이해할
   수 있는 비유(예: 7=불닭볶음면 수준)가 함께 표시된다.

### User Story 2 - 메뉴판 스캔 → 개인화 위험도 안내 (Priority: P1)

사용자가 식당에서 한식 메뉴판을 촬영(또는 갤러리에서 업로드)하면, 메뉴가 사용자 언어로
번역되고, 각 메뉴 옆에 자신의 식이 제약 기준 위험도(안전/주의/위험)가 표시된다. 메뉴를
선택하면 음식 상세가 열린다.

**Why this priority**: 기존 번역/지도 앱과의 핵심 차별점이자 제품의 킬러 기능. 안전(헌법 III)과
직결된다.

**Independent Test**: 프로필이 설정된 사용자가 메뉴판 사진을 올리면, 번역된 메뉴 목록과 각
항목의 위험도 표시가 나오는 것으로 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 프로필이 설정된 사용자, **When** 카메라 탭에서 메뉴판을 촬영하거나 갤러리에서
   사진을 업로드하면, **Then** 스캔 진행 상태가 표시되고 결과로 메뉴 항목들이 촬영 이미지 위
   오버레이(파파고/구글렌즈식)로 사용자 언어로 번역되어 나타난다.
2. **Given** 번역 오버레이 결과, **When** 사용자가 토글 버튼을 누르면, **Then** 원본 메뉴판
   이미지와 번역 오버레이가 서로 전환된다.
3. **Given** 번역된 메뉴 항목, **When** 결과가 표시되면, **Then** 각 메뉴 우측에 사용자 제약
   기준 위험도(안전/주의/위험)가 디자인된 아이콘으로 표시된다(기본 이모지 사용 금지, 헌법).
4. **Given** 위험도가 표시된 메뉴, **When** 사용자가 한 메뉴를 선택하면, **Then** 음식 상세가
   하단 시트(bottom sheet) 형태로 열린다.
5. **Given** 인식/번역 신뢰도가 낮은 항목, **When** 위험도를 판정할 근거가 불충분하면,
   **Then** '안전'으로 단정하지 않고 '주의' 이상으로 표시한다(헌법 III, 보수적 처리).

### User Story 3 - '주의' 성분 → 사장님께 확인 (Priority: P2)

음식 상세에서 '주의'로 분류된 재료/성분을 사용자가 누르면, 왜 주의인지 설명과 함께, 사장님께
보여줄 한국어 확인 화면을 띄울 수 있다.

**Why this priority**: 매장별로 달라지는 성분의 불확실성을 사용자가 직접 해소하게 하는, 안전
원칙의 실질적 안전장치. US2가 선행되어야 한다.

**Independent Test**: 음식 상세에서 '주의' 성분을 눌러 한국어 확인 문구(메뉴명+성분 질문)가
정확히 표시되는 것으로 단독 검증 가능.

**Acceptance Scenarios**:

1. **Given** 음식 상세를 연 사용자, **When** '주의'로 표시된 성분을 누르면, **Then** "이 메뉴에는
   해당 성분이 N% 들어가며, 매장별로 다를 수 있으니 사장님께 확인하라"는 안내와 버튼이 나온다.
2. **Given** 위 안내 화면, **When** 사용자가 확인 버튼을 누르면, **Then** 장소 언어(MVP=한국어)로
   "{메뉴명}에 {성분}이 들어가나요?"가 크게, 그 아래 "알러지가 있어 위험하다"는 설명이 표시된다.
3. **Given** 사장님 확인 화면, **When** 표시되는 메뉴명을 보면, **Then** 그 메뉴명은 메뉴판에서
   인식된 메뉴명과 정확히 일치한다(헌법 III).

---

### User Story 4 - 음식 탐색 & 리뷰 (Priority: P2)

사용자가 음식 탭에서 음식을 둘러보고, 음식별 상세 정보(설명·사진·평점)와 리뷰를 보고, 같은
국적 사용자들의 평점/리뷰만 모아보거나 자신의 언어로 번역해 읽으며, 직접 평점·리뷰를 남긴다.

**Why this priority**: '음식 단위' 정보 공유라는 제품 정체성(헌법 II)을 완성하지만, 안전 핵심
루프(US1~US3)보다는 후순위.

**Independent Test**: 음식 상세에서 리뷰 목록 조회, 같은 국적 필터, 번역, 리뷰 작성이 각각
동작하는 것으로 검증 가능.

**Acceptance Scenarios**:

1. **Given** 음식 탭, **When** 사용자가 진입하면, **Then** 음식들을 탐색하기 쉬운 형태로 보여주고
   각 음식의 상세로 이동할 수 있다.
2. **Given** 음식 상세, **When** 표시되면, **Then** 사용자 언어 메뉴명, 전체 평점과 같은 국적
   평점, 한국인 실사용 느낌의 설명(영어 기준 150자 이내), 음식 사진, 90% 이상 들어가는
   재료/성분을 위험/주의/안전 순으로 보여준다.
3. **Given** 리뷰 목록, **When** 사용자가 리뷰를 보면, **Then** 각 리뷰에 작성자 국적(국기)과
   랭킹이 표시되고, 누르면 사용자 언어로 번역되며, 같은 국적 리뷰만 모아볼 수 있다.
4. **Given** 음식 상세, **When** 사용자가 평점과 리뷰를 작성하면, **Then** 저장되어 해당 음식의
   리뷰 목록과 평점 집계에 반영된다.

### User Story 5 - 프로필 관리 & 랭킹 (Priority: P3)

사용자가 자신의 정보를 조회/수정/삭제하고, 자신이 쓴 리뷰를 모아 보고, 유쾌한 랭킹(입문자 ~
뼛속까지 한국인)을 확인한다.

**Why this priority**: 유지/재방문 가치를 주지만 핵심 안전 루프 이후에 필요.

**Independent Test**: 프로필 수정, 내 리뷰 목록 조회, 랭킹 표시가 동작하는 것으로 검증 가능.

**Acceptance Scenarios**:

1. **Given** 로그인 사용자, **When** 프로필에서 식이 정보/국적/언어 등을 수정하면, **Then**
   변경이 저장되고 이후 위험도 판정에 반영된다.
2. **Given** 프로필, **When** 사용자가 '내 리뷰'를 보면, **Then** 자신이 작성한 리뷰가 모여
   표시된다.
3. **Given** 프로필, **When** 랭킹을 보면, **Then** 활동에 따른 유쾌한 등급명이 표시된다.

---

### User Story 6 - 홈: 개인화 허브 (Priority: P3)

식당이 아닌 곳에서 앱을 연 사용자가, 홈에서 최근 스캔/조회한 음식을 다시 보고 리뷰를 남기도록
유도받고, 자신의 프로필 기준 안전도가 표시된 추천 음식을 발견하며, 메뉴판 스캔으로 빠르게
진입한다.

**Why this priority**: 재방문·리뷰 수집·발견을 돕는 진입점이지만, 핵심 안전 루프(US1~US3)
이후의 가치다. 기존 데이터만 재사용하므로 가볍게 구현 가능.

**Independent Test**: 홈 진입 시 최근 본 음식 목록, 추천 음식, 스캔 CTA가 표시되고 각 항목에서
음식 디테일/리뷰/카메라로 이동되는 것으로 검증 가능.

**Acceptance Scenarios**:

1. **Given** 스캔/조회 이력이 있는 사용자, **When** 홈에 진입하면, **Then** 최근 스캔/조회한
   음식이 표시되고 "리뷰 남기기"로 유도된다.
2. **Given** 홈, **When** 추천 음식이 표시되면, **Then** 각 음식에 사용자 프로필 기준 위험도가
   함께 표시된다.
3. **Given** 홈, **When** 사용자가 스캔 CTA를 누르면, **Then** 카메라(메뉴판) 화면으로 이동한다.
4. **Given** 이력이 없는 신규 사용자, **When** 홈에 진입하면, **Then** 빈 상태 안내와 함께 스캔/
   탐색을 권유한다.

### Edge Cases

- 메뉴판이 흐리거나 일부만 인식될 때: 인식 실패/저신뢰 항목은 위험도를 단정하지 않고 '주의'
  이상으로 표시하거나 재촬영을 안내한다.
- 메뉴판에 한식이 아닌/번역 불가 항목이 섞일 때: 번역 가능한 항목만 처리하고 나머지는 원문과
  함께 "판정 불가"로 구분한다.
- 사용자가 다중 알러지/제약을 가질 때: 가장 높은 위험도를 대표 표시하되, 상세에서 제약별 근거를
  확인할 수 있어야 한다.
- 위험도 판정에 필요한 성분 데이터가 없는 음식: '안전'으로 표시하지 않는다(보수적 기본값).
- 리뷰가 아직 없는 음식 / 같은 국적 리뷰가 없는 경우: 빈 상태를 명확히 안내한다(0건 표기).
- 오프라인/네트워크 불안정: 스캔·번역 실패 시 명확한 오류와 재시도를 제공한다.
- 미성년/민감정보: 종교·건강(알러지) 정보는 민감정보로 취급한다(헌법 IV, 추가 제약).

## Requirements *(mandatory)*

### Functional Requirements

**온보딩 & 프로필**

- **FR-001**: 시스템은 이메일 인증, 애플 로그인, 구글 로그인으로 회원가입/로그인을 제공해야 한다.
- **FR-002**: 시스템은 가입 시 닉네임, 국적, 사용 언어를 필수로 수집해야 한다.
- **FR-003**: 시스템은 알러지·비건/채식·종교 기반 못 먹는 음식을 구체적으로 입력받되, 온보딩에서
  **건너뛸 수 있다**(입력 강력 권장, 추후 프로필에서 채우도록 유도). 단, 제약이 **미입력 상태**인
  사용자에게는 개인화 위험도 '안전'을 **절대 표시하지 않으며**, '주의' 또는 '판정불가'로만 표시한다
  (false-safe 금지, 헌법 III·SC-003). 미입력 사용자에게는 제약 입력 시 개인화가 활성화됨을 고지한다.
- **FR-004**: 시스템은 매운맛 허용도를 0~10 단계로 **선택** 입력받고, 각 단계에 외국인 친화적
  비유를 함께 제시해야 한다.
- **FR-005**: 시스템은 관심/먹어본 한식을 선택 입력받되, 인기순으로 9개를 먼저 보여주고 선택 시
  연관 항목을 추가로 펼치는 방식을 제공해야 한다.
- **FR-006**: 사용자는 자신의 프로필(식이 정보 포함)을 조회·수정·삭제할 수 있어야 한다.
- **FR-032**: 계정 삭제 시 시스템은 개인·민감정보(알러지·종교 등)를 즉시 완전 삭제해야 한다.
  단, 사용자가 작성한 리뷰는 작성자를 익명화하여 보존하며(음식 평점 집계 유지), 익명화 이후
  해당 리뷰에서 개인 식별이 불가능해야 한다.

**메뉴판 스캔 & 번역**

- **FR-007**: 사용자는 카메라로 메뉴판을 촬영하거나 갤러리에서 사진을 업로드할 수 있어야 한다.
- **FR-008**: 시스템은 메뉴판에서 메뉴 항목을 인식하고 사용자 언어(reader language)로 번역해
  **촬영한 메뉴판 이미지 위에 오버레이 형태**(파파고/구글렌즈식)로 표시하되, 각 항목 우측에
  위험도를 함께 표시해야 한다(FR-011 연계).
- **FR-037**: 사용자는 결과 화면에서 **원본 메뉴판 이미지와 번역 오버레이를 토글**로 전환할
  수 있어야 한다. 오버레이 인식/정렬이 불완전할 때 원본과 대조해 확인할 수 있어야 한다
  (안전 보조, 헌법 III).
- **FR-031**: 시스템은 인식된 메뉴 항목을 큐레이션 DB의 **표준 음식 카탈로그**에 매핑(정규화)해
  음식 단위로 집계해야 한다. 카탈로그에 매칭되는 표준 음식이 없으면 '미등록 음식'으로 구분하고,
  위험도를 '안전'으로 단정하지 않는다(FR-013·FR-015 연계).
- **FR-033**: '미등록 음식'에 대해 시스템은 (a) 번역은 표시하되, (b) 위험도를 안전/주의/위험으로
  단정하지 않고 **'판정불가'** 상태로 표기하며, (c) "안전 판정 불가, 사장님께 확인" 경고를
  노출해야 한다. 또한 (d) 미등록 음식 발생을 백엔드에 로그로 기록해 추후 카탈로그 보강에
  활용할 수 있어야 한다.
- **FR-009**: 시스템은 스캔/처리 중 진행 상태를 사용자에게 표시해야 한다.
- **FR-010**: 카메라 탭은 메뉴판/가공식품/음식 모드 전환 UI를 갖되, **MVP에서는 메뉴판 모드만
  동작**하고 나머지는 비활성/추후 안내로 표시한다.

**위험도 판정 (안전)**

- **FR-011**: 시스템은 각 메뉴에 대해 사용자 프로필 기준 위험도를 안전/주의/위험 3단계로
  산출해 메뉴 목록과 상세에 표시해야 한다. 판정 근거 데이터가 없는 '미등록 음식'은 '판정불가'
  상태로 표기한다(FR-033).
- **FR-012**: 위험도 판정의 근거(어떤 성분/재료/확률에 기반했는지)는 추적 가능해야 하며, 근거
  없이 '안전'으로 단정해서는 안 된다.
- **FR-013**: 판정에 필요한 정보가 불충분하거나 신뢰도가 낮으면 시스템은 위험을 낮추는 방향으로
  ('주의' 이상으로) 처리해야 한다.
- **FR-014**: 시스템은 음식의 재료/성분 중 90% 이상 포함되는 항목을 위험/주의/안전 순으로
  상세에 표시해야 한다.
- **FR-015**: 위험도 판정은 **자체 큐레이션 데이터베이스**(사람이 검수한 음식–재료/성분 및
  알러지/종교/식이 매핑)를 권위 있는 출처로 사용해야 한다. 이 DB는 별도 팀이 구축하며, 본
  명세·설계는 **해당 DB가 제공된다고 가정**한다(구체 스키마/적재 방식은 본 기능 범위 밖). 검수된
  데이터가 없는 음식은 '안전'으로 표시하지 않는다(FR-013과 연계, 보수적 기본값).

**음식 상세 & 사장님 확인**

- **FR-016**: 음식 상세는 사용자 언어 메뉴명, 전체 평점과 같은 국적 평점, 한국인 실사용 느낌의
  설명(영어 기준 150자 이내), 음식 사진, 위험도순 재료/성분을 화면 위→아래 순서로 보여줘야 한다.
- **FR-017**: 사용자가 '주의' 성분을 선택하면 시스템은 그 성분이 들어가는 비율과 매장별 변동
  가능성을 설명하고, 사장님 확인 화면으로 가는 버튼을 제공해야 한다.
- **FR-018**: 사장님 확인 화면은 장소 언어(MVP=한국어)로 "{메뉴명}에 {성분}이 들어가나요?"를
  크게, 그 아래에 알러지 위험 설명을 표시해야 한다.
- **FR-019**: 사장님 확인 화면의 메뉴명은 메뉴판에서 인식된 메뉴명과 정확히 일치해야 한다.

**음식 탐색 & 리뷰**

- **FR-020**: 사용자는 음식 탭에서 음식을 탐색하고 각 음식 상세로 이동할 수 있어야 한다.
- **FR-021**: 시스템은 음식별로 전체 평점과 같은 국적 사용자 평점을 5점 별점 평균으로 집계해
  표시해야 한다.
- **FR-022**: 사용자는 음식에 평점(5점 별점, 1~5 정수)과 리뷰 본문을 작성할 수 있어야 한다.
- **FR-023**: 리뷰 목록은 각 리뷰에 작성자 국적(국기)과 랭킹을 표시하고, 사용자 언어로 번역할
  수 있어야 하며, 같은 국적 리뷰만 모아보는 필터를 제공해야 한다.

**프로필 & 랭킹**

- **FR-024**: 사용자는 자신이 작성한 리뷰를 모아 볼 수 있어야 한다.
- **FR-025**: 시스템은 활동 기반의 유쾌한 랭킹 등급(예: 입문자 ~ 뼛속까지 한국인)을 산출해
  프로필과 리뷰에 표시해야 한다. MVP의 등급 점수는 **복합 점수 = 리뷰 수 + 리뷰한 음식
  다양성(서로 다른 음식 종류 수) + 스캔 활동**에 가중치를 적용해 계산한다(구체 가중치/구간은
  설계 단계에서 확정). 2차에서 커뮤니티 활성도를 가중 요소로 추가한다.

**네비게이션 & 공통**

- **FR-026**: 앱은 홈/음식/카메라(가운데 강조)/커뮤니티/프로필 5탭과 헤더(좌측 로고, 우측 알림·
  검색)를 제공한다. 단 커뮤니티 탭은 MVP에서 자리만 두고 내용은 제공하지 않는다.
- **FR-034**: 홈은 사용자의 최근 스캔/조회한 음식을 표시하고 리뷰 작성으로 유도해야 한다.
- **FR-035**: 홈은 추천 음식을 사용자 프로필 기준 위험도와 함께 표시해야 한다.
- **FR-036**: 홈은 메뉴판 스캔으로 바로 이동하는 CTA를 제공하며, 이력이 없는 사용자에게는
  스캔/탐색을 권유하는 빈 상태를 보여줘야 한다.
- **FR-027**: 사용자에게 노출되는 모든 UI 문자열은 i18n 리소스로 관리되어야 하며 단일 언어로
  하드코딩되어서는 안 된다(헌법 I). MVP는 reader 언어 **9개**(`en`, `zh-Hans`, `zh-Hant`, `ja`,
  `vi`, `id`, `th`, `ru`, `es`)를 모두 제공하며, 앱 폰트는 라틴·CJK·태국·키릴 스크립트를 모두
  렌더할 수 있어야 한다(글자 깨짐 0).
- **FR-028**: 장소에 묶인 데이터(메뉴·성분·사장님 확인 문구)는 장소 언어를 `locale` 등으로
  식별 가능하게 모델링하되, MVP에서 장소 언어 값은 한국어(`ko`)로 고정한다(헌법 I).
- **FR-029**: 위험도 신호를 포함한 모든 아이콘/시각 요소는 기본(시스템/유니코드) 이모지가 아닌
  디자인된 아이콘 세트로 구현해야 한다(헌법 추가 제약).
- **FR-030**: 시스템은 위험도 표시가 참고 정보이며 최종 확인 책임이 사용자/매장에 있음을
  고지해야 한다. 고지 수위는 **온보딩 시 1회 명시적 동의 + 위험도/상세 화면에 상시 고지 문구**
  표시로 한다. 동의 이력은 보관한다.

### Key Entities *(데이터 관여)*

- **User(사용자)**: 계정 식별자, 인증 수단(이메일/애플/구글), 닉네임, 국적, reader 언어, 매운맛
  허용도, 랭킹 등급.
- **DietaryRestriction(식이 제약)**: 사용자에 속한 알러지/종교/비건·채식 등 못 먹는 항목.
  민감정보로 취급. 계정 삭제 시 즉시 완전 삭제 대상(FR-032).
- **Food(음식/메뉴)**: 음식 단위의 정보 집계 대상. **큐레이션 DB의 표준 음식 카탈로그가 음식
  ID의 단일 기준**이다. 표준 메뉴명, 설명(언어별), 대표 사진, 평점 집계. 스캔된 메뉴 항목은
  이 표준 음식으로 정규화되어 매핑되며, 카탈로그에 없는 항목은 '미등록 음식'으로 구분한다.
  '식당'이 아니라 '음식'이 1차 단위(헌법 II).
- **Ingredient/Component(재료/성분)**: 음식에 포함되는 재료·성분과 포함 비율, 알러지/종교/식이
  태그 매핑.
- **FoodRiskAssessment(위험도 판정)**: 특정 사용자×음식의 안전/주의/위험/판정불가 결과와 판정
  근거(어떤 제약·성분·비율 기반인지). '판정불가'는 미등록 음식에 사용. 추적 가능해야 함(FR-012).
- **MenuScan(스캔)**: 촬영/업로드한 메뉴판 1건과 인식·번역 결과(메뉴 항목 목록, 신뢰도).
- **Review(리뷰)**: 사용자×음식의 평점(5점 별점, 1~5 정수)과 본문, 작성자 국적·랭킹, 원문 언어.
- **OwnerConfirmation(사장님 확인 문구)**: 메뉴명+성분 질문을 장소 언어(MVP=ko)로 생성한 표시
  콘텐츠.
- **Locale/SupportedLanguage(언어)**: reader 언어 목록 = **MVP 9개 전부 지원**
  (`en`, `zh-Hans`, `zh-Hant`, `ja`, `vi`, `id`, `th`, `ru`, `es`; BCP-47). place 언어(MVP=ko).
  스크립트 커버리지: 라틴·CJK(중/일)·태국·키릴. (Session 2026-06-29)
- **RankingTier(랭킹 등급)**: 복합 활동 점수(리뷰 수 + 음식 다양성 + 스캔 활동) 기준 등급
  정의(입문자 ~ 뼛속까지 한국인). 2차에서 커뮤니티 활성도 가중 추가 예정.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 신규 사용자가 가입부터 식이 프로필 입력 완료까지 5분 이내에 마칠 수 있다.
- **SC-002**: 사용자가 메뉴판을 촬영한 뒤 위험도가 표시된 번역 결과를 보기까지 평균 10초 이내다.
- **SC-003**: 위험도가 '안전'으로 표시된 메뉴 중 실제로 사용자의 등록 제약과 충돌하는 비율
  (위험한 오안전, false-safe)이 0%에 수렴한다(안전 최우선, 헌법 III). 불확실 항목은 '주의'로
  분류되어 이 지표를 악화시키지 않는다.
- **SC-004**: 메뉴판 스캔 사용자의 80% 이상이 적어도 하나의 메뉴 상세를 열어본다(핵심 가치
  도달).
- **SC-005**: 음식 상세를 본 사용자의 30% 이상이 같은 국적 리뷰 또는 사장님 확인 기능 중
  하나를 사용한다(차별 기능 활용).
- **SC-006**: 메뉴판 메뉴명과 사장님 확인 화면의 메뉴명 불일치 발생률 0%.
- **SC-007**: 1차 MVP가 2026-07-10까지 위 핵심 흐름(US1~US3)을 사용 가능한 상태로 제공한다.

- 사용자는 스마트폰과 카메라, 인터넷 연결을 사용할 수 있다.
- **MVP는 reader 언어 9개를 모두 지원**한다: `en`, `zh-Hans`, `zh-Hant`, `ja`, `vi`, `id`, `th`,
  `ru`, `es`. 정적 UI는 9개 i18n 리소스, 동적 콘텐츠는 BE 서버측 번역. 폰트는 라틴+CJK+태국+키릴
  커버. i18n 미설계·하드코딩은 불가(헌법 I). (Session 2026-06-29 확정)
- **회원가입은 필수**다. 개인화 위험도 판정이 프로필에 의존하므로 게스트(비로그인) 스캔은 MVP
  범위에서 제외한다.
- **위험도/성분 데이터는 별도 팀이 구축하는 자체 큐레이션 DB가 제공된다고 가정**한다. 본 기능은
  그 DB를 소비하며, DB 구축/스키마 자체는 범위 밖이다(통합 지점은 `/speckit-plan`에서 정의).
- 위험도 표시는 의료적 보장이 아니라 참고용 안내이며, 온보딩 1회 동의 + 상시 고지로 안내하고,
  최종 확인은 사장님 확인 흐름으로 보완한다(FR-030).
- 다중 제약 사용자의 메뉴 목록 위험도는 "가장 높은 위험도" 대표 표기를 기본으로 한다(상세에서
  제약별 근거 확인 가능).
- 같은 국적 평점/리뷰가 부족한 초기(콜드 스타트)에는 0건/부족 상태를 명시적으로 표기한다.
- 지도·식당 엔티티, 가공식품/음식 직접 인식, 커뮤니티 게시판, 푸시 리마인드는 MVP 범위 밖이다
  (헌법 V 단계별 출시).
- 기술 스택은 미정이며 `/speckit-plan`에서 결정한다. 본 명세는 구현 방식을 규정하지 않는다.
